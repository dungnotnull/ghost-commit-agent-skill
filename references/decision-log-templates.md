# Decision Log Templates

_Load during Phase 3 when generating decision logs and PR descriptions._

---

## Decision Log — Standard Template

```markdown
# Decision: {{title}} — {{date}}

**Commit:** `{{short_hash}}` ({{full_hash}})
**Author:** {{author_name}} <{{author_email}}>
**Branch:** `{{branch_name}}`
**Files changed:** {{file_count}} files, +{{insertions}} −{{deletions}} lines
**Risk level:** {{risk_level}} <!-- Low / Medium / High -->
**Change type:** {{commit_type}}

---

## Context

{{context_section}}
<!-- What situation, bug, requirement, or constraint prompted this change?
     Source: .ghost-context "why" section, or inferred from diff -->

## Decision

{{decision_section}}
<!-- What was decided and implemented.
     Source: diff analysis — what was actually added/changed/removed -->

## Alternatives considered

{{alternatives_section}}
<!-- What other approaches were tried or considered, and why they were rejected.
     Source: .ghost-context "what tried" section.
     If absent: "Not documented — use .ghost-context before committing to capture this." -->

## Constraints

{{constraints_section}}
<!-- What technical, business, or external constraints drove the approach.
     Source: .ghost-context "constraints" section. -->

## Future implications

{{implications_section}}
<!-- What does this decision mean for future work?
     TODOs left behind, tech debt incurred, follow-up PRs needed, 
     areas that will need updating if requirements change.
     Source: diff TODOs/FIXMEs + inference -->

## References

{{references_section}}
<!-- Issue numbers, PRs, Slack threads, documentation, ADRs.
     Source: .ghost-context "references" section + branch name parsing -->

---

_Generated by ghost-commit v1.0.0 | {{timestamp}}_
_Context quality: {{context_quality}} <!-- HIGH / MEDIUM / LOW -->_
```

{{context_section}}

{{decision_section}}

## References

{{context_section}}

{{decision_section}}

{{alternatives_section}}

## Constraints

{{constraints_section}}

## PR Description Template

```markdown
## What does this PR do?

{{pr_summary}}
<!-- 1–3 sentences. What is the user-facing or system-level effect of this change?
     NOT "changed X to Y" — "Users can now..." or "Fixes the bug where..." -->

## Why?

{{pr_why}}
<!-- The motivation. Why was this necessary? What problem does it solve?
     Source: .ghost-context "why" section -->

## Changes

{{pr_changes}}
<!-- Bullet list of the key changes. Focus on what matters for review,
     not every line changed. Group by area:
     - auth/: Added PKCE flow support
     - session/: Fixed TTL unit calculation
     - tests/: Added integration tests for both scenarios -->

## How to test

{{pr_testing}}
<!-- Specific steps to verify this works:
     1. Create a session
     2. Wait 30 seconds
     3. Verify session is still active (was expiring after 30ms before) -->

## Checklist

- [ ] Tests pass locally
- [ ] No regressions in related functionality
- [ ] Documentation updated (if applicable)
- [ ] Breaking changes documented (if applicable)
{{extra_checklist_items}}

## Breaking changes

{{pr_breaking}}
<!-- "None" or description of what breaks and migration path -->

## Screenshots / recordings

{{pr_screenshots}}
<!-- "N/A" or actual screenshots for UI changes -->

## References

{{pr_references}}
<!-- Closes #N, Refs #N, ADR links, Slack threads -->
```

## Context Quality Labels

Include one of these at the bottom of every decision log:

```
Context quality: HIGH
Source: .ghost-context file with all sections populated.
```

```
Context quality: MEDIUM
Source: .ghost-context file partially populated. Some sections inferred from diff.
Sections inferred: [list]
```

```
Context quality: LOW
Source: Pure diff analysis — no .ghost-context provided.
To improve: Run `python ghost_commit.py --context` before committing to add context.
```

## Section Population Rules

### Context section
Priority:
1. `.ghost-context` → "Why I'm making this change" verbatim
2. Branch name parsing: `fix/JIRA-123-session-timeout` → "Fixing JIRA-123: session timeout"
3. Recent commit messages: infer project context from last 5 commits
4. Diff analysis: "This change modifies the session management subsystem to..."

### Decision section
Always from diff — describe what was implemented:
- For `fix`: "Corrected [X] by [Y]"
- For `feat`: "Added [X] which allows [Y]"
- For `refactor`: "Restructured [X] by [Y] — behavior unchanged"

### Alternatives section
Priority:
1. `.ghost-context` → "What I tried first" verbatim
2. If absent: `"Not documented. Use .ghost-context 'What I tried first' section to capture this."`

Never fabricate alternatives — only write what the developer told you.

### Implications section
Infer from diff:
- TODO/FIXME comments added → list them
- Temporary workaround patterns (`# HACK`, `# temporary`) → flag
- Migration files → note related migration PR needed
- Large dependency added → note maintenance overhead
- Config changes → note what needs updating in other environments

## Filename Convention

```
.decisions/YYYY-MM-DD-<short-hash>.md
```

Examples:
```
.decisions/2026-06-07-a3f9c12.md
.decisions/2026-06-07-bb823f1.md
```

Sorting by date is natural when listing directory contents.

## .gitignore Recommendation

Add to project `.gitignore`:
```gitignore
# ghost-commit decision logs (developer context — not for version control)
.decisions/
.ghost-context
.ghost-commit-cache/
```

Or commit them — some teams find decision logs valuable in the repo:
```gitignore
# Keep decision logs in repo
# .decisions/ is NOT gitignored
```

The `install_hook.py` script asks which preference during setup.
